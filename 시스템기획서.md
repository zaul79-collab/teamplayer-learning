## 1. 시스템 개요

### 목적

팀플레이어 3덕목(겸손·갈망·영리함) 학습 경험을 **콘텐츠 관리자 → 학습자 → 멘토** 세 주체가 각자의 역할에 맞게 활용하는 통합 학습 시스템.

### 핵심 설계 원칙

| 원칙 | 내용 |
|------|------|
| 반복은 시스템에 | 콘텐츠 업데이트, 퀴즈 교체, 일정 알림은 시스템이 처리 |
| 판단·코칭은 사람에게 | 학습 데이터를 멘토에게 제공해 인간적 개입의 질을 높임 |
| 내재적 동기 설계 | 점수 경쟁이 아닌 자기이해와 성장을 목표로 UX 설계 |
| 점진적 공개 | 모든 콘텐츠를 한 번에 노출하지 않고 완료 흐름에 따라 순차 개방 |
| 데이터 기반 운영 | 이탈·완료율·오답 패턴으로 콘텐츠를 지속 개선 |

## 3. 지식데이터 레이어 — 어드민

### 3-1. 관리 대상 데이터

| 데이터 유형 | 항목 | 설명 |
|-------------|------|------|
| 모듈 콘텐츠 | 제목, 설명, 학습 목표, 카드슬라이드(N장), 체크리스트 | 7개 모듈 |
| 연습 퀴즈 | 문항, 선택지(4개), 정답, 해설 | 모듈별 3문항 |
| 게이트 퀴즈 | 문항, 선택지(4개), 정답 | 모듈별 5문항 |
| 자기진단 설문 | 문항, 척도 레이블, 덕목 분류 | 고정 18문항 |
| 재인증 문항 | Day+7/+21 각 3문항, 자기보고 여부 | 착근기 검증용 |
| UX 카피 | 버튼 레이블, 안내 문구, 피드백 텍스트 | 운영 중 수정 가능 |

### 3-2. 어드민 기능 명세

**콘텐츠 관리**

| 기능 | 설명 |
|------|------|
| 모듈 CRUD | 모듈 생성·수정·삭제, 공개/비공개 상태 전환 |
| 카드슬라이드 편집 | 슬라이드 순서 변경, 텍스트/이미지 편집, 장 수 조정 |
| 퀴즈 문항 관리 | 문항 추가·수정·삭제, 정답 변경, 난이도 태그 |
| 설문 문항 관리 | 항목 수정, 덕목 재분류, 척도 레이블 변경 |
| 버전 관리 | 수정 이력 보존, 이전 버전 복구 |

**학습 설정 관리**

| 기능 | 설명 |
|------|------|
| 착근기 일정 | Day+7/+21 기준일 설정 (인증일 기준 자동 계산) |
| 게이트 통과 기준 | 통과 점수 설정 (현재 설계: 5/5 만점) |
| 재도전 제한 | 게이트 재도전 횟수 제한 on/off, 제한 횟수 입력 |
| 콘텐츠 공개 방식 | 순차 개방 vs. 전체 공개 전환 설정 |

**코호트 & 사용자 관리**

| 기능 | 설명 |
|------|------|
| 코호트 등록 | 그룹명, 시작일, 소속 부서 |
| 학습자 등록 | 이름, 소속, 코호트, 멘토 배정 |
| 멘토 등록 | 담당 코호트, 알림 수신 채널 설정 |
| 어드민 권한 | 콘텐츠 편집권과 사용자 관리권 분리 |

### 3-3. 핵심 데이터 모델

```
Module
  ├── id, title, virtue (h/g/s), order, status (draft/published)
  ├── description, objectives[]
  ├── Cards[]
  │     ├── id, order, title, body, type (text/checklist/image)
  │     └── CheckItems[]
  ├── PracticeQuestions[3]  ── explanation 포함
  └── GateQuestions[5]      ── explanation 없음 (게이트는 정오답 미표시)

Question
  ├── id, text, options[4], correctIndex
  └── explanation (연습 퀴즈 전용)

SurveyItem
  ├── id, text, virtue, order
  └── scaleLabels: { 3: '언제나', 2: '때때로', 1: '거의 안 함' }

RecertSession
  ├── day (7 | 21), title, subtitle
  ├── Questions[3]  ── selfReport 여부
  └── PracticeItems[3]
```

## 4. 백오피스 레이어 — 학습 로직 & API

백오피스는 지식데이터를 학습자에게 안전하게 전달하고, 학습 행동을 올바르게 판정·저장·집계하는 엔진이다.

### 4-1. 콘텐츠 전달 API

| 엔드포인트 | 기능 |
|-----------|------|
| `GET /modules` | 전체 모듈 목록 (학습자 진행 상태 포함) |
| `GET /modules/:id` | 특정 모듈 상세 (카드, 퀴즈 포함) |
| `GET /survey` | 자기진단 18문항 |
| `GET /recert/:day` | 재인증 문항 (Day+7 or Day+21) |

**보안 처리 포인트**: 게이트 퀴즈의 정답(correctIndex)은 API 응답에서 제외하고, 채점 요청 시 서버 측에서만 비교.

### 4-2. 학습 로직 엔진

**모듈 순차 개방 로직**

```
기본 설정 (순차 개방):
  M1 완료(게이트 통과) → M2 개방
  M2 완료 → M3 개방
  ...
  M5 완료 → M6 개방
  M6 완료 → M7 개방 (전 모듈 순차 개방 — 기본값)

자유 개방 설정 시:
  전체 모듈 처음부터 개방
  → 어드민에서 on/off 전환
```

**게이트 채점 로직**

```
POST /gate/submit
  body: { moduleId, answers[5] }

서버 처리:
  1. 모듈에 해당하는 GateQuestions 조회
  2. answers vs. correctIndex 비교
  3. score = 정답 수
  4. pass = (score >= 통과 기준)  ← 어드민 설정값 참조
  5. 결과 저장 → GateAttempt 레코드
  6. pass이면 모듈 완료 처리 → 다음 모듈 개방

응답: { score, pass, attemptCount }
```

**자기진단 채점 로직**

```
POST /survey/submit
  body: { answers: { itemId: score } }  ← score: 1~3

서버 처리:
  1. 덕목별 점수 합산 (h: 6문항 × 최대 3점 = 18점)
  2. 각 덕목 퍼센테이지 계산
  3. 집중 포인트 = 점수 최저 덕목
  4. 저장 → SurveyResult 레코드

응답: { h, g, s } (각 점수)
```

**착근기 스케줄러**

```
인증 완료 시각 기록 → certifiedAt
  certifiedAt + 7일 → Day+7 버튼 활성화 트리거
  certifiedAt + 21일 → Day+21 버튼 활성화 트리거

스케줄러 실행:
  매일 0시: certifiedAt 기준 D+7, D+21 도래 학습자 조회
           → 해당 학습자 상태 업데이트
           → 알림 발송 큐 추가
```

### 4-3. 세션 & 인증 관리

| 기능 | 설명 |
|------|------|
| 학습자 인증 | SSO 연동 (사내 계정) 또는 초대 링크 방식 |
| 세션 토큰 | JWT, 만료 7일, 학습 상태 포함 |
| 권한 분리 | 학습자 / 멘토 / 어드민 토큰 분리 |
| 정답 보호 | 게이트 정답은 서버에서만 처리, 클라이언트 미전달 |

### 4-4. 알림 발송 엔진 (상세는 §9)

| 트리거 | 발송 대상 | 채널 |
|--------|-----------|------|
| Day+7/+21 도래 | 학습자 | 이메일 |
| 7일 이상 비활동 | 학습자, 멘토 | 이메일 |
| 게이트 3회 미통과 | 멘토 | 이메일 |
| 전 모듈 완료 | 학습자, 멘토, 어드민 | 이메일 |

## 5. 사용자인터페이스 레이어 — 학습자 UX

### 5-1. 설계 근거 — 5가지 이론 적용

#### ① 자기결정이론 (Self-Determination Theory, Deci & Ryan)

내재적 동기는 자율성·유능감·관계성이 충족될 때 발생한다.

| 욕구 | 설계 반영 |
|------|-----------|
| **자율성** | 모듈 순서 자유 선택, 언제든 재진단 가능, "다시 풀기" 제공 |
| **유능감** | 연습 퀴즈 즉각 정오답 + 해설, 게이트 통과 시 명확한 성취 표현 |
| **관계성** | 멘토 연결 구조, 코호트 단위 학습, 콘텐츠 내 "우리" 언어 |

#### ② 포그 행동 모델 (Fogg Behavior Model)

행동 = 동기 × 능력 × 촉발(Trigger).

| 요소 | 설계 반영 |
|------|-----------|
| **동기** | 덕목 색상 시각화, 성장 타임라인, 비판단적 진단 결과 |
| **능력** | 카드 1장씩, 퀴즈 3~5문항, 모바일 원핸드 최적화 |
| **촉발** | Day+7/+21 버튼 자동 활성화, 체크리스트 완료 시 CTA 자동 노출 |

#### ③ 차이카르니크 효과 (Zeigarnik Effect)

미완성 과제는 기억에 더 오래 남는다.

| 설계 반영 |
|-----------|
| `N / 18 완료` — 항상 미완 상태 인지 |
| 모듈 목록 비완료 dot — 채워지지 않은 상태 시각 유지 |
| 게이트 `선택 전` 배지 — 미선택 항목이 눈에 띔 |
| 타임라인 점선 항목 — "아직 안 된 것"을 지속 노출 |

#### ④ 행동 유도 (Affordance) & 마찰 설계

| 행동 | 설계 |
|------|------|
| 다음 학습으로 이동 | CTA 버튼이 콘텐츠 완료 시 자동 등장 (마찰 최소화) |
| 게이트 미준비 통과 | 연습 퀴즈를 먼저 노출 → 게이트는 별도 탭 (준비 유도) |
| 자기진단 이탈 | 18문항을 덕목별 6개 그룹화 → 한 덕목씩 완료감 |
| 게이트 미선택 제출 | "선택 전" 배지 시각화 → 제출 시 해당 카드로 이동 |

#### ⑤ 피크엔드 법칙 (Peak-End Rule)

경험은 절정 순간과 마지막 순간으로 기억된다.

| 지점 | 설계 |
|------|------|
| **절정 (Peak)** | 게이트 통과 결과 바텀시트 — 점수 강조, 성취 메시지, 다음 모듈 CTA |
| **마지막 (End)** | 재인증 완료 화면 — 격려 메시지 + "여정으로 돌아가기" |

### 5-2. UX 카피라이팅 원칙

| 원칙 | 규칙 | 이론 근거 |
|------|------|-----------|
| 동사로 시작 | 풀기, 시작하기, 채점하기 | 행동 촉발 명확화 (Fogg) |
| 목적어 명시 | ~~시작하기~~ → 게이트 채점하기 | 인지 명료화 |
| 진행 표시어 | `N / N 완료`, `선택 전`, `✓ 선택됨` | Zeigarnik 완료 욕구 |
| 긍정 프레이밍 | `~입니다` — `~아닙니다` 시작 금지 | 심리적 수용성↑ |
| 화살표(→) 규칙 | 다음 화면 이동 시만 사용 | 방향성 일관성 |

### 5-3. 화면별 핵심 UX 의도

| 화면 | 핵심 UX 의도 | 주요 장치 |
|------|-------------|-----------|
| 홈 | 현재 상태 인식 + 학습 재진입 유도 | 진행 현황 숫자, 빠른 진입 CTA |
| 모듈 목록 | 경로 조망 + 선택 자율성 | 덕목별 섹션, 완료 dot |
| 학습 탭 | 소화 가능한 분량 + 능동적 확인 | 카드뉴스 1장씩, 체크리스트 |
| 연습 퀴즈 | 즉각 피드백으로 유능감 | 즉시 정오답 + 해설 |
| 게이트 | 진지한 확인 + 성취 절정 | 전체 제출 방식, 통과 바텀시트 |
| 자기진단 | 심리적 안전 + 자기이해 | 비판단 안내, 덕목별 그룹화 |
| 나의 여정 | 성장 서사 + 지속 동기 | 타임라인, 착근기 재인증 |

### 5-4. 온보딩 플로우 화면 (미구현)

```
[온보딩 Step 1] 서비스 소개
  "팀플레이어가 된다는 것은 무엇일까요?"
  → 3덕목 Venn 다이어그램 소개

[온보딩 Step 2] 자기진단 유도
  "먼저 지금의 나를 확인해볼까요?"
  → 자기진단 화면으로 이동

[온보딩 Step 3] 학습 경로 안내
  "진단 결과를 바탕으로 학습을 시작해보세요"
  → 모듈 목록 화면으로 이동
```

### 6-1. 수집 데이터 항목

### 6-2. 학습자 뷰 — 완성도 & 동기부여

**홈 화면 진행 현황 카드**

```
겸손  갈망  영리함
 1/3   0/2   0/2
```

숫자가 채워질수록 성취감 누적. 덕목 색상으로 즉각 인식.

**나의 여정 타임라인**

- 완료 항목: 날짜 + 채워진 dot — "내가 여기까지 왔다"는 서사
- 미완 항목: 점선 dot + 흐린 텍스트 — 앞으로 갈 방향 조망
- 덕목 완성 마일스톤(🎯): 해당 덕목 마지막 모듈 게이트 통과 시 강조 표시 — 게이트 자체는 모듈별 7개
- 착근기 항목: 완료 시 날짜 기록 + dot 채워짐

**자기진단 결과 뷰**

- 덕목별 점수 바 차트 — 숫자보다 시각적 분포로 자기이해 유도
- 집중 포인트 카드 — "이 덕목을 먼저 키워보세요" (처방 아닌 제안)
- 학습 완료 후 재진단 비교 — 사전·사후 변화 확인 (예정)

**성취 배지 시스템 (예정)**

| 배지 | 획득 조건 |
|------|-----------|
| 겸손 완성 | M1~M3 게이트 모두 통과 |
| 갈망 완성 | M4~M5 게이트 모두 통과 |
| 영리함 완성 | M6~M7 게이트 모두 통과 |
| 팀플레이어 | 전 모듈 완료 |
| 착근 완료 | Day+21 재인증 완료 |
| 꾸준한 학습자 | 7일 연속 학습 (예정) |

## 9. 알림 & 넛지 시스템

학습자의 자율성을 침해하지 않는 선에서 학습 지속을 돕는 적시 알림.

### 9-1. 학습자 알림

| 알림 유형 | 트리거 | 채널 | 메시지 방향 |
|-----------|--------|------|-------------|
| 학습 재개 넛지 | 3일 비활동 | 이메일 | "지난번 학습에서 멈췄어요" |
| 지연 경고 | 7일 비활동 | 이메일 | "코호트 동료들이 앞서가고 있어요" |
| Day+7 도래 | certifiedAt+7일 | 이메일 | "1차 재인증이 열렸어요" |
| Day+21 도래 | certifiedAt+21일 | 이메일 | "마지막 착근 확인을 해볼까요?" |
| 게이트 통과 축하 | 게이트 통과 즉시 | 앱 인앱 | "통과! 다음 모듈로 가볼까요?" |

### 9-2. 멘토 알림

| 알림 유형 | 트리거 | 채널 |
|-----------|--------|------|
| 멘티 학습 이탈 | 담당 멘티 7일 비활동 | 이메일 |
| 게이트 반복 미통과 | 동일 게이트 3회 미통과 | 이메일 |
| Day+7 재인증 미완료 | D+10일 경과 | 이메일 |
| 멘티 전체 완료 | 전 모듈 게이트 통과 | 이메일 |

### 9-3. 넛지 원칙

- 동일 알림은 최대 주 1회 (과잉 알림 방지)
- 학습자가 알림 수신 여부 직접 설정 가능
- 멘토 알림은 담당 코호트 내 멘티에만 한정

## 10. 콘텐츠 운영 체계

### 10-1. 콘텐츠 업데이트 정책

| 유형 | 처리 방식 |
|------|-----------|
| 오탈자/문구 수정 | 즉시 반영 |
| 퀴즈 문항 교체 | 신규 코호트부터 적용, 기존 코호트 현행 유지 |
| 모듈 구조 변경 | 버전 분기, 진행 중인 학습자 완료 후 신규 버전 적용 |
| UX 카피 변경 | 즉시 반영, 변경 이력 어드민에서 확인 |

### 10-2. 오답 패턴 기반 콘텐츠 개선 프로세스

```
주간 오답률 리포트 (어드민)
  │
  ├── 오답률 50% 이상 문항 → 콘텐츠 담당자에게 알림
  │     → 문항 난이도 또는 문구 검토 → 수정 후 다음 코호트 적용
  │
  └── 특정 카드에서 이탈 집중 → 해당 카드 체류 시간 분석
        → 콘텐츠 길이/난이도 조정 검토
```

## 11. 역할 및 권한

| 역할 | 접근 가능 기능 | 데이터 범위 |
|------|---------------|-------------|
| **어드민** | 콘텐츠 CRUD, 코호트·사용자 관리, 전체 현황 | 전체 |
| **멘토** | 코호트 현황, 개인 상세, 알림 수신 | 담당 코호트 학습자 |
| **학습자** | 모듈 학습, 퀴즈, 자기진단, 재인증, 나의 여정 | 본인만 |

### 권한 분리 원칙

- 학습자는 타 학습자 데이터 접근 불가
- 멘토는 담당 코호트 외 접근 불가
- 진단 결과는 학습자 동의 하에 멘토 공개 (기본값: 공개)
- 게이트 정답은 서버에서만 처리, 어드민만 조회 가능
