# 팀플레이어 학습 플랫폼 UX 원칙

> 기능 설계·콘텐츠 작성·개발 방향 결정 시 참조한다.
> 새 기능을 만들거나 기존 기능을 변경할 때 이 원칙에 어긋나지 않는지 먼저 확인한다.

---

## 1. UX 관점 5대 원칙

### 1-1. 인지 부하 최소화
**근거**: Cognitive Load Theory (Sweller)  
한 번에 처리할 수 있는 정보량에는 한계가 있다. 과부하가 오면 학습이 아닌 생존 모드가 된다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 퀴즈 | 문항 1개씩 표시, 보기 최대 4개, 타이머 없음 |
| 자기진단 | 선택지를 숫자 척도 대신 텍스트 3개로 단순화 |
| 공통 | 화면 내 장식·불필요 정보 제거, 다음 할 일 하나만 표시 |

---

### 1-2. 즉각적 피드백
**근거**: Feedback Loop (Nielsen Norman Group)  
행동과 결과 사이의 시간이 길수록 학습 효과가 급감한다. 피드백은 즉시, 명확하게.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 연습 퀴즈 | 선택 즉시 정답 색상 변경 + 해설 카드 표시 |
| 게이트 | 전체 제출 후 틀린 항목 하이라이트 + 해설 |
| 자기진단 | 제출 후 즉시 "개발 초점" 요약 표시 |

---

### 1-3. 진행도 가시성
**근거**: Zeigarnik Effect, Progress Principle  
미완성 과제는 기억에 남는다. 진행 중임을 눈에 보이게 하면 완료 동기가 높아진다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 홈 화면 | M1~M7 각 모듈 완료 상태 한눈에 표시 |
| 퀴즈 | "4 / 8 문항" 상단 진행바 표시 |
| 착근기 | Day+7, Day+21 체크인 남은 일수 표시 |

> ⚠ 진행바는 "남은 것"이 아닌 **"완료한 것"** 기준으로 표시한다. 압박감 대신 성취감을 준다.

---

### 1-4. 오류 허용 설계
**근거**: Error Tolerance (Don Norman)  
실수를 막는 것보다 실수해도 안전한 환경이 학습자의 심리적 안전을 만든다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 연습 퀴즈 | 기록 없음 · 횟수 제한 없음 · 틀려도 부정적 메시지 없음 |
| 게이트 | 재시도 가능 · "다시 도전하세요"가 아닌 중립 언어 사용 |
| 자기진단 | "약점"이 아닌 "개발 초점" 언어 사용 |

---

### 1-5. 일관된 패턴
**근거**: Consistency & Standards (Nielsen 10 Heuristics)  
패턴이 예측 가능하면 인지 부하가 줄고, 학습자는 인터페이스가 아닌 내용에 집중할 수 있다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 모든 모듈 | 동일한 레이아웃 순서: 읽기 → 연습 → 게이트 |
| 공통 UI | 버튼 위치·색상·텍스트 전 화면 통일 |
| 퀴즈 결과 | 정답=초록, 오답=빨강 — 화면 전체에서 동일 |

---

## 2. 자기주도학습 관점 5대 원칙

### 2-1. 자율성
**근거**: Self-Determination Theory (Deci & Ryan)  
외부 강제가 아닌 스스로 선택한다는 느낌이 내재적 동기를 만든다. 통제는 동기를 죽인다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 공통 | 학습 속도 자율 · 중간 저장 후 나중에 이어하기 가능 |
| 자기진단 | 비채점 · 결과 공개 여부 본인 선택 |
| 공통 | 타이머·데드라인 없음 (착근기 알림 제외) |

---

### 2-2. 인출 연습 효과
**근거**: Testing Effect (Roediger & Karpicke)  
다시 읽는 것보다 기억에서 꺼내는 행위 자체가 기억을 강화한다. 퀴즈가 곧 학습이다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 연습 퀴즈 | 모듈 읽기 직후 바로 연습 — 복습이 아닌 학습 자체 |
| 학습 흐름 | "공부하고 풀기"가 아닌 "풀면서 공부하기" UX 흐름 |
| 착근기 | Day+7 재인증 = 인출 연습 타이밍 설계 |

---

### 2-3. 간격 반복
**근거**: Spaced Repetition (Ebbinghaus)  
한 번 몰아 학습보다 시간 간격을 두고 반복하는 것이 장기 기억에 훨씬 효과적이다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 착근기 | Day+7, Day+21 재인증이 간격 반복 구현 |
| 착근기 알림 | 망각 곡선 직전 타이밍에 리마인드 노출 |
| 게이트 재시도 | 다른 변형 문항 출제 = 변형 간격 연습 |

---

### 2-4. 메타인지
**근거**: Metacognition (Flavell)  
"내가 무엇을 알고 모르는지"를 아는 능력. 자기 학습을 조절하는 핵심 역량이다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 자기진단 | 18문항(언제나·때때로·거의 아님) = 자신의 덕목 수준 스스로 파악 |
| 자기진단 결과 | 점수 대신 "어디를 더 볼지" 언어로 제시 |
| 착근기 | Day+21 자기진단 재방문 = 변화 성찰 구조화 |

> ⚠ **자기진단은 '평가'가 아니라 '성찰 도구'** — 점수로 사람을 줄세우지 않는다.
> - 18문항 응답 후 가장 낮은 덕목 1개만 **'개인 개발 초점'**으로 제시한다. (등수·합격선·타인 비교 없음)
> - 해석 참고 밴드(덕목별 6문항·6~18점): **16~18 강점 / 13~15 점검 / 6~12 개발**. 이는 줄세우기가 아닌 자기 해석을 돕는 참고선이다.
> - 유형 이름(7유형)은 **자기인식·코칭용**이며, 실재 동료에게 꼬리표로 붙이지 않는다. (윤리 가드레일)

---

### 2-5. 실행 의도
**근거**: Implementation Intention (Gollwitzer)  
"이런 상황에서 이렇게 한다"는 if-then 형식이 실제 행동 전이율을 3배 높인다.

| 적용 대상 | 설계 결정 |
|-----------|----------|
| 캡스톤 | if-then 실행의도 작성 항목이 이 원리 구현 |
| 착근기 | Day+7 체크인 = 실행의도 실천 여부 확인 |
| 게이트 | 사례 분류 문항 = 실제 상황 적용 연습 |

---

## 3. 원칙 충돌 지점 — 미리 합의된 결정

기능 설계 시 아래 충돌 사례를 참고해 일관성을 유지한다.

### 진행도 가시성 vs 자율성
- 충돌: 진행바가 "몇 % 남았다"는 압박이 될 수 있다.
- **결정**: 진행 표시는 하되 "남은 것" 대신 "완료한 것"을 강조. (예: "3개 완료" O / "4개 남음" X)

### 즉각 피드백 vs 인출 연습
- 충돌: 너무 빠른 정답 노출은 인출 시도 자체를 방해한다.
- **결정**: 연습 퀴즈는 선택 후 0.5초 딜레이 후 피드백 표시.

### 게이트 100% 요구 vs 오류 허용
- 충돌: 핵심 모듈 100% 통과 기준이 학습자에게 "시험"처럼 느껴질 수 있다.
- **결정**: 재시도 UX를 "연습하고 다시 오세요" 흐름으로 자연스럽게 연결. 불합격 메시지 금지.
- **게이트 미통과는 처벌이 아니라 지원 신호로 다룬다.** 1·2회 미통과는 격려 메시지로 응대하고, 3회째에는 멘토에게 도움 메시지(이메일)를 자동 발송해 사람이 개입한다. **반복은 시스템이, 코칭은 사람이.**

## 4. 언어 가이드 — 쓰면 안 되는 말 vs 써야 하는 말

| 쓰면 안 되는 말 | 써야 하는 말 | 이유 |
|----------------|-------------|------|
| 틀렸습니다 | 다시 확인해볼까요 | 오류 허용 |
| 점수: 60점 | 이 항목들을 더 살펴보세요 | 메타인지·낙인 방지 |
| 약점 | 개발 초점 | 성장 마인드셋 |
| 4개 남음 | 3개 완료 | 자율성·성취감 |
| 재시험 | 다시 도전 | 심리적 안전 |
| 당신은 ○○ 유형입니다 | ○○ 항목에서 성장 기회가 보입니다 | 낙인 금지 (윤리 가드레일) |

## 5. 보상 설계 원칙

> 핵심 방향: **"보상"보다 "기록"**. 배지 모으기 게임이 아닌, 내가 걸어온 여정을 돌아보는 경험.

### 왜 과도한 게이미피케이션은 안 되는가
자기결정이론(SDT)의 과잉정당화 효과 — 외부 보상이 "원래 하고 싶었던 것"을 "보상 받으려는 것"으로 바꿔버린다. 겸손·갈망·영리함을 다루는 이 시스템에서 배지 모으기 게임처럼 느껴지면 콘텐츠의 진지함이 희석된다.

---

### 5-1. 순간 메시지 (Milestone Message)

의미 있는 순간에만 등장. 모든 행동에 반응하지 않는다.

| 발생 시점 | 메시지 방향 |
|-----------|------------|
| 모듈 완료 | 배운 내용이 아닌, 생긴 것을 표현 |
| 게이트 통과 | 통과 사실이 아닌, 무엇이 확인됐는지 |
| 게이트 재시도 | 격려 · 다음 단계 안내 · 절대 실패 언어 없음 |
| 캡스톤 인증 | 출발선 메시지 (완성이 아닌 시작) |
| 착근기 Day+21 완료 | 변화를 스스로 돌아볼 수 있는 질문형 마무리 |

**언어 원칙**: "완료했습니다" 대신 성장 서사를 담은 문장

| 쓰면 안 되는 말 | 써야 하는 말 |
|----------------|-------------|
| M2 완료했습니다 ✓ | 겸손에 대한 자신만의 언어가 생겼습니다 |
| 게이트를 통과했습니다 | 7가지 유형을 실제 상황에 적용할 수 있게 됐습니다 |
| 인증 완료 | 이상적인 팀 플레이어 여정을 시작했습니다 |
| 21일을 완료했습니다 | 3주 전의 나와 지금의 나, 어떤 점이 달라졌나요? |

---

### 5-2. 인증서 (Certificate)

- 발급 시점: 캡스톤 통과 시 **단 한 번**
- 형태: 다운로드 가능한 PDF
- 문구 방향: "이상적인 팀 플레이어 여정을 시작했습니다" — 완성이 아닌 출발선
- 점수·등수·비교 정보 없음

---

### 5-3. 개인 성장 타임라인 (Personal Timeline)

경쟁·비교 없이 **나의 여정 기록**만 남긴다.

- 각 모듈 완료일, 캡스톤 인증일, 착근기 완료일이 시간순으로 표시
- 예시 항목: "6월 11일 — M2 겸손 완료", "6월 18일 — 캡스톤 인증", "6월 25일 — Day+7 체크인"
- 타인의 타임라인은 볼 수 없음 (개인 전용)
- 매달 새로운 항목이 쌓이는 구조 → "매달 취득" 개념을 기록으로 구현

---

### 5-4. 절대 하지 않는 것

| 금지 항목 | 이유 |
|-----------|------|
| 점수 기반 배지 | 인성 점수화 금지 (윤리 가드레일 E1) |
| 등수·순위 공개 | 낙인·비교 금지 (윤리 가드레일 E4) |
| 타인과 비교 | 자율성·심리적 안전 침해 |
| 스트릭(연속 학습일) 압박 | 자율성 침해, 불안 유발 가능 |
| "이달의 팀 플레이어" 선정 | 경쟁·비교 구조 |

## 6. 여정 시각화 — 팀플레이어 벤 다이어그램

> 핵심 방향: 세 원이 처음엔 분리된 채 각각 성장하다가, 덕목이 완성될수록 서로 가까워지고 합쳐져 중심에 "이상적인 팀플레이어" 영역이 만들어진다.

### 6-1. 원 구조

| 원 | 덕목 | 색상 |
|----|------|------|
| 왼쪽 | 겸손 | 노랑 |
| 오른쪽 | 갈망 | 파랑 |
| 아래 | 영리함 | 초록 |

- 홈 화면에 항상 노출 — 학습자가 매번 자신의 현재 상태를 체감
- 공통 모듈은 세 원의 교차 중심부에 해당하는 개념 (이상적인 팀플레이어 전반)

### 6-2. 4단계 서사 구조

| 단계 | 조건 | 시각 변화 |
|------|------|----------|
| 1단계 | 초기 상태 | 세 원이 분리, 작고 불규칙한 유기체 형태, 연한 색으로 각각 존재 |
| 2단계 | 모듈 완료마다 | 크기가 커지고 형태가 점점 둥글어지며 색이 진해짐 |
| 3단계 | 덕목별 게이트 전체 통과 | 완벽한 원이 되어 중심을 향해 이동 시작 |
| 4단계 | 세 원 모두 완료 | 교차 영역 형성 → 중심 "이상적인 팀플레이어" 빛남 |

**원 성장 메카닉**: 크기 4단계 (작음 → 중간 → 크게 → 최대), 형태는 유기체 블롭에서 완벽한 원으로 수렴. 덕목 이름 텍스트는 초기에 숨김 상태로 시작해 원이 커질수록 자연스럽게 나타남.

- 점수·퍼센트 표시 **금지** — 형태와 색감으로만 표현
- "○○% 완료" 같은 수치 노출 없음 (인성 점수화 금지, 윤리 가드레일 E1)
- 덕목마다 다른 타이밍에 이동 — 각각의 완성이 살아있는 느낌을 줌

### 6-3. 이동 트리거

- **덕목별 게이트 전체 통과 시** 해당 원이 중심을 향해 이동
- 세 덕목이 각각 다른 타이밍에 완료되어 이동 → 순차적 수렴 서사
- 이동은 애니메이션으로 표현 (갑자기 바뀌는 것이 아닌 서서히)

### 6-4. 완성 화면 (단 한 번)

캡스톤 통과 순간에만 등장하는 전환 화면.

| 단계 | 내용 |
|------|------|
| 전환 | 세 원이 완전히 합쳐지며 중심이 빛나는 애니메이션 |
| 문구 | "팀플레이어로서의 여정이 시작됐습니다" |
| 다음 | 인증서 PDF 다운로드로 자연스럽게 연결 |

- "완성됐습니다"가 아닌 **출발선 문구** — 보상 설계 원칙(5-1)과 일치
- 이 화면은 학습 전체를 통틀어 단 한 번만 등장

### 6-5. 구현 단계

| 단계 | 내용 |
|------|------|
| 1단계 (MVP) | 세 원 분리 정적 표시, 완료/미완료 2상태 (연한색 → 풀색) |
| 2단계 | 덕목 완료 시 원 이동 애니메이션 추가 |
| 3단계 | 단계적 opacity 변화 + 완성 전환 화면 |

## 7. 모션 원칙

> 핵심 방향: 모션은 **시선 유도·맥락 이해를 돕는 도구**일 뿐, 장식이 아니다. 과하지 않게, 그리고 `prefers-reduced-motion`을 존중한다.

| 적용 대상 | 모션 역할 |
|-----------|----------|
| CTA 버튼 | 은은한 광택 스윕·호흡으로 다음 행동 유도 (비활성 버튼은 효과 제거) |
| 다음 문항 버튼 | 화살표 넛지로 클릭 대기 상태에서 시선 유도 |
| 홈 히어로 | 유기적 배경(orb drift)으로 살아있는 느낌만 부여 |
| 모델 소개 | 벤 다이어그램 세 원의 순차 등장 → 세 덕목이 합쳐지는 서사를 시각으로 전달 |

- 모든 장식 모션은 `@media (prefers-reduced-motion:reduce)`에서 비활성화한다 (접근성·전정기관 민감 사용자 보호).
- 모션이 정보를 *대체*하지 않는다 — 애니메이션이 꺼져도 내용·상태는 그대로 전달돼야 한다.
